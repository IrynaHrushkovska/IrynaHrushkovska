git config --global user.name blabla - внесення змін на всьому ПК щодо встановлення ім'я користувача, яке буде відображатись під логами коммітів для можливості відслідковування ПІБ того, хто вніс відповідні зміни

git config --global user.email blabal@bla- внесення змін на всьому ПК щодо встановлення e-maol користувача, що буде відображатись в логах поряд з ім'ям для можливості коммунікації в подальшому

git init - команда для створення локального репозиторію (на ПК)

git status - показує статус відслідковування файлів, які відслудковуються, які ні

git add - команда для додавання файлу для відслідковування (git add . - додасть всі папки до відслдковування)

git commit - створення комміту/фіксації внесених змін в код, за якими потрібно слідкувати, або від них відштовхуватись перед написанням додаткової частини коду

git log - видає перелік існуючих коммітів з хешкодами та ПІБ+email користувача, що створив комміт/p>

git checkout - команда для вибору гілки, в якій планується робота

git branch - команда для переіменування та створення нових гілок, в такому форматі git branch - покаже доступні/створені гілки

git push - відправляє на віддалений репозиторій (на github або інший хостінг IT-проектів) зміни в коді

git pull - дозволяє отримати зміни коду, що були зроблені у віддаленому репозиторії

git merge - дозволяє об'єднати 2 гілки коду в один проект

git clone - дозволяє скопіювати до редактору коду проект з віддаленого репозиторію

.gitignore - це файл, в якому розробник може прописати перелік тих файлів/артефактів/зображень, які не мають ігноруватисі git-ом , тобто не мають передаватись до репозиторію. Корисно для зручності використання додання файлів до відслідковування командою git add . (тобто додативсе), але при цьому будуть проігноровані (не додані) ті файли які вказані у файлі .gitignore

git revert - дозволяє відміти комміт без його видалення з історії, тобто фактично ми повернемось в стан до внесених змін які, наприклад призвели до багів, але створимо цим новий наступний комміт (комміт в якому дописана частина коду яка призвела до багів залишиться в історії)

git reset - важко для розуміння, але як зрозуміла - команда для повернення до попереднього стану коду посилаючись на конкретний комміт, одна з опцій - reset --hard, поверну код до попереднього стану з видаленням всіх наступних, створених версій

git rebase - робить злиття 2х гілок для трохи за іншим принципом. При використанні rebase - історія коммітів є більш "чистою", лінійною. Приходить на допомогу, коли потрібно об'єднати код своєї гілки з кодом іншого розробника або своєї дочірньої, але не вносити ці зміни в головну функціональну вітку проекту

git cherry-pick - дозволяє виконати часткове злиття коду між гілками (виконати злиття вибірково вказаних коммітів іншої гілки)

git stash - дозволяє зберегти написану частину коду в сховище для можливості дописати код через час і закомітити вже повноцінний дописаний функціонал (треба/хочу відкласти виконання задачі яку вже частково вирішила, але не хочу щоб хтось бачив/знав що частина зроблена або для більш чистої історії комітів (не створювати зайвих))
